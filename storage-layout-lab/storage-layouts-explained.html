Storage engines · from first principles

Same bytes. Three ways to arrange them.

A row store, a column store, and an LSM partition store hold the exact same table. The arrangement — not the code, not the speed of the machine — decides which queries fly and which ones fall off a cliff.

Postgres · ClickHouse · Cassandra — one mental model for all three.

Postgres — B-tree row store
ClickHouse — columnar MergeTree
Cassandra — LSM, partition-addressed
The one question underneath all of it

How many bytes must I read to answer this query?

Every storage engine is a bet on that single number. A disk (or even RAM) reads in blocks, not individual values. So the real cost of a query isn't "how many rows match" — it's how many bytes the engine is forced to drag off storage to find them. The ratio of bytes-read to bytes-you-actually-wanted has a name:

Read amplification You asked for 20 bytes. The engine read 4 KB to get them. That's 200× read amplification — and it's the difference between 400ms and 6 seconds on the same data.

A storage layout is just a decision about what to keep physically next to what, so that the queries you care about read mostly bytes they need, and the queries you don't care about pay the amplification instead. There is no free arrangement. Every layout is fast along one grain and slow across it. The three engines below made three different bets.

Layout 01

Row store + B-tree — organized around the row

Postgres keeps each whole row contiguous on disk. All of row 1's columns sit together, then all of row 2's, and so on. To find a specific row fast, it builds a B-tree index: a shallow sorted tree that lets you jump from 20 million rows to the exact one in about four hops. The bet: you'll usually want a whole record, found by its key.

What it optimizes Jump-to-one-record and change-it-in-place. The B-tree makes point lookups nearly flat as the table grows (depth grows logarithmically). The cost lives on the other side: a scan that needs only amount still drags the fat payload off disk on every single row, because the row is the unit of storage.
point lookup 1.37ms · but full-column aggregate 5,871ms (paying for payload it never uses)
Layout 02

Columnar MergeTree — organized around the column

ClickHouse stores each column contiguously instead: every amount value together, every payload together. A query that sums amount reads only the amount column and never touches payload at all. Same values sitting side by side also compress far better. The bet: you'll scan many rows but only a few columns.

The point-lookup twist ClickHouse has no per-row index. Its "primary index" is sparse — one marker per granule of ~8,192 rows. A lookup by key finds the right granule, then reads and scans the whole granule to find one row. So a "point" read still touches thousands of rows — bounded and cheap, but never as sharp as a B-tree. This is the single prediction most people get wrong: they expect either instant (like Postgres) or scan-slow. It's neither.
aggregate 415ms (~14× faster than Postgres) · point lookup 9.9ms — respectable, never scan-slow
The catch nobody sees coming The sort order is fixed at table creation (the ORDER BY key). You cannot add a B-tree later to make a new column fast — a filter on an unplanned column is always a full column scan. Fast because columnar, but there's no escape hatch to point-read.
Layout 03

LSM, partition-addressed — organized around the write

Cassandra optimizes a different thing entirely: never make a write wait for a disk seek. Writes append to memory, get flushed later as immutable files, and reads find data by hashing the partition key straight to its location. The bet: a firehose of writes, keyed by something natural, read back by that same key — and it must scale past one machine.

cassandra Writes never seek
memtable in RAM → flush to immutable SSTable → compaction merges later
the LSM write path · why it's a write firehose
Writes go to memory first — that's why they never wait on a disk seek.
What it optimizes Write throughput and horizontal scale. Appending to memory means writes never seek; the price is paid later, at read time (a key may live in several SSTables) and at compaction. Add nodes and write throughput grows roughly linearly — the single-node number understates it.
highest sustained random-key writes (~7,200/s, single node) · point read by key 3.56ms
The cliff — no graceful slow mode Every other layout degrades off its grain: a Postgres scan is slow but runs; a ClickHouse filter is a scan but returns. Cassandra doesn't degrade off the partition key — it refuses or fails. A cross-partition aggregate took 4.2 minutes client-side; a filter on a non-key column errored outright. If you need a second access path, you build a second table. You don't add an index.
Reasoning it out

Which grain does your query cut along?

You don't memorize a table — you reason from one question per axis. Each engine wins where the query runs along the grain its bytes are organized around, and loses where it cuts across.

The question you ask about the workloadReach forWhy (from the lab)
Fetch whole records by a known key, most of the time? Postgres 1.37ms — B-tree lands on the row; nothing else approximates it.
Scan millions of rows but only a few columns (SUMs, dashboards)? ClickHouse 415ms vs 5,871ms — reads only the columns named, compressed.
Write firehose keyed by device/user/session, read back by that key, must outgrow one node? Cassandra ~7,200/s and scales linearly by adding nodes.
People will filter on columns you didn't plan for? Postgres CREATE INDEX turns a 6.7s scan into 4ms. The only late escape hatch.
Writes arrive strictly row-by-row, no batching possible? not ClickHouse ~400/s row-by-row — every insert creates a storage part. Batch, or buffer with Kafka.
Provisioning tight or bursty? Postgres Under a 512MB cap: PG spilled to disk and held; ClickHouse OOM-crashed.
Mixed, unknown, or evolving — you're not sure yet? Postgres The default until a measurement proves you need to move off it.

Along the grain

The query reads mostly the bytes it needs. Postgres fetching a row by id. ClickHouse summing one column. Cassandra reading by partition key. This is the fast path each engine was built to serve — often orders of magnitude ahead of the others.

Across the grain

The query fights the layout. Postgres dragging payload through a column scan. ClickHouse filtering an unplanned column. Cassandra crossing partitions. Postgres and ClickHouse degrade here; Cassandra falls off a cliff — slow, then impossible.

The interview version

Each engine organizes its bytes around a different thing, and that choice decides everything downstream:

A row store organizes bytes around the row.

Wins when you want whole records by key. Loses when you scan millions of rows for two columns — it drags the whole row off disk to read a sliver of it.

A column store organizes bytes around the column.

Wins scans and aggregates by an order of magnitude, and stays acceptable at point reads via a sparse granule index. But every insert creates a storage part, so write cost only amortizes in batches — and the sort order is frozen at table creation.

An LSM partition store organizes bytes around the partition key.

Writes never seek and scale linearly across nodes. But any access path that crosses partitions doesn't slow down gracefully — it fails.

The flip trigger is never the engine's quality. It's whether the query cuts along the layout's grain or across it. Default to Postgres; reach for ClickHouse when the workload is provably "few columns, many rows, batchable writes"; reach for Cassandra when it's provably "one partition key, write-heavy, multi-node."
